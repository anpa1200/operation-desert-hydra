# Security Model — Operation Desert Hydra

This document describes what the lab does, what it does not do, and what safeguards are in place.

---

## Lab Design Principles

**Defensive scope only.** The lab validates whether a detection stack captures the right telemetry events for simulated behaviors. It is not a red-team exercise, an offensive capability demonstration, or a malware execution environment.

**Benign payloads only.** Every simulation in `lab/ansible/playbooks/validate.yml` uses:
- Standard Windows system tools (`powershell.exe`, `wscript.exe`, `schtasks.exe`, `cmd.exe`, `certutil.exe`)
- Synthetic file content that produces the target Sysmon event without malicious execution
- Known-safe third-party binaries (`ScreenConnect.exe` installer, `nslookup.exe`)
- No shellcode, no live malware, no reverse shells, no real C2

**Isolated lab network.** The Windows 10 VM runs on VirtualBox NAT. It has no inbound connectivity from the internet. All outbound connections from the VM go through the NAT gateway (`10.0.2.2`) and are limited by VirtualBox's NAT rules.

---

## What the Lab Does Not Do

| Not present | Why |
|-------------|-----|
| Live malware execution | Simulations use benign equivalents that produce the same Sysmon event type |
| Real C2 connections | No Mori, BugSleep, or other MuddyWater malware runs in this lab |
| Credential exfiltration | LSASS dump simulation creates a `.dmp` file for EID 11; the file is deleted immediately after event confirmation |
| Telegram bot API connections | Simulation targets EID 3 event generation; the VM uses NAT and does not connect to api.telegram.org |
| Privilege escalation chains | Ansible runs via WinRM with a provisioned administrator account for provisioning purposes only |
| Lateral movement | The lab is single-VM by design |

---

## `.dmp` File Handling

Step 31 (`det_mw_0010`, Rule C) simulates LSASS credential access by creating a file named `lsass_test.dmp`. The Ansible task that creates this file is immediately followed by a cleanup task that deletes it. The file:
- Contains no real LSASS memory
- Is created using `Out-File`, not `MiniDumpWriteDump` or any credential dumping tool
- Is deleted within the same playbook run, before the run completes
- Is visible as `step-31c-det-mw-0010-rule-c.png` in the validation proofs (EID 11 FileCreate event captured before deletion)

## Attribution Language Guardrails

Public sources widely report MuddyWater / Seedworm as Iran-linked activity associated with MOIS. This project does not independently verify or endorse that attribution. All ATT&CK technique mappings in this project are analyst candidates based on published source language, not independent attribution findings.

---

## Data Handling

- No real IOCs are generated by the lab — all events are synthetic
- No data leaves the lab VM except via Winlogbeat → Elasticsearch (localhost only)
- The `.env` file (credentials) is excluded from the repo via `.gitignore`; only `.env.template` is tracked
- No real credentials or tokens are committed to the repository
